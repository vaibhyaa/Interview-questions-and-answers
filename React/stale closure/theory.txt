xplain stale closures in depth

Stale closures are one of those topics that separate “I can use React” from “I understand React.” Let’s go deep but keep it practical.

🧠 First: What is a Closure? (quick refresh)

A closure means a function “remembers” variables from the scope where it was created.

function App() {
  let count = 0;

  function log() {
    console.log(count);
  }

  return log;
}

👉 log remembers count

⚠️ What is a Stale Closure?

A stale closure happens when a function captures old (outdated) state/props instead of the latest value.

👉 In React:

Every render creates a new version of variables, but closures may still hold old ones

🔥 Core Problem (VERY IMPORTANT)

👉 React re-renders → creates NEW state values
👉 But your function still uses OLD values from when it was created

💣 Example 1: Classic Stale Closure
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    setInterval(() => {
      console.log(count);
    }, 1000);
  }, []);

  return <button onClick={() => setCount(count + 1)}>+</button>;
}
❗ What happens?
useEffect runs once (because [])
Closure captures count = 0
Interval keeps logging 0 forever

👉 Even if UI shows 5, 10, 20…

💥 Why This Happens

Because of this:

useEffect(() => { ... }, []);

👉 Effect runs only once
👉 Closure is created only once
👉 It “freezes” the initial value

✅ Fix #1: Add Dependencies
useEffect(() => {
  const id = setInterval(() => {
    console.log(count);
  }, 1000);

  return () => clearInterval(id);
}, [count]);

✔ Now effect re-runs with latest count

⚠️ But this recreates interval every time → not always ideal

✅ Fix #2: Functional State Update (BEST for counters)
setCount(prev => prev + 1);

👉 Avoids stale state inside updates

✅ Fix #3: useRef (Advanced & Real-world)
const countRef = useRef(count);

useEffect(() => {
  countRef.current = count;
}, [count]);

useEffect(() => {
  const id = setInterval(() => {
    console.log(countRef.current);
  }, 1000);

  return () => clearInterval(id);
}, []);

✔ Interval runs once
✔ Always gets latest value
✔ No re-creation

💣 Example 2: Event Listener Bug
useEffect(() => {
  function handleClick() {
    console.log(count);
  }

  window.addEventListener("click", handleClick);

  return () => window.removeEventListener("click", handleClick);
}, []);

👉 Same problem → always logs old count

💣 Example 3: Async / API Calls
useEffect(() => {
  async function fetchData() {
    const res = await fetch(`/api?q=${query}`);
    const data = await res.json();
    console.log(query);
  }

  fetchData();
}, []);

👉 Uses old query → wrong API call

🧠 Mental Model (Interview Gold)

👉 Each render = new snapshot of state

👉 Closures = photo taken at that time

👉 If you don’t update dependencies → you keep using old photo

⚠️ When Stale Closures Commonly Happen
setInterval / setTimeout
Event listeners
API calls
Promises
useEffect with []
Memoized callbacks (useCallback)
🔥 Rules to Avoid Stale Closures
1. Always respect dependency array

👉 If you use it → add it

2. Use functional updates when possible
setState(prev => ...)
3. Use useRef for mutable latest value

👉 Especially for:

timers
subscriptions
4. Don’t “hack” dependencies
// ❌ BAD
// eslint-disable-next-line

👉 This causes bugs in production

🚨 Interview Trap Question

👉 “Why is my state not updating inside setInterval?”

Correct answer:

Because of stale closure — the callback captures the initial state due to empty dependency array.
